# TextField

TextField er et inputfelt for kortere tekst eller tall, som kontonummer, e-postadresse eller belĆøp.

## Egnet til

- Kortere tekstinput (Ć©n linje)
- Input med klart forventet format

## Uegnet til

- Lengre fritekst. Bruk `TextArea` i stedet.
- Komplekse eller sammensatte input (f.eks. dato med flere felt)
- Valg fra forhĆ„ndsdefinerte alternativer. Bruk `select`, `radio` eller `checkbox`.

## Retningslinjer

### Tydelige labels gir forutsigbarhet

Alle `TextField` skal ha en synlig og beskrivende label. Labelen skal forklare hva brukeren skal skrive inn, ikke hva de skal gjĆøre. "E-postadresse" er bedre enn "Skriv inn e-post".

NĆ„r labelen beskriver selve dataen, kan brukeren raskt orientere seg uten Ć„ lese hele setninger. Dette er spesielt viktig i skjemaer med mange felter, der god struktur reduserer kognitiv belastning og risiko for feil.

### UnngĆ„ bruk av placeholder

Placeholder forsvinner nĆ„r brukeren begynner Ć„ skrive, kan forveksles med utfylt innhold, og er ikke tilstrekkelig erstatning for label eller description. Bruk `description` for eksempler og hint ā det er alltid synlig.

`description="11 siffer, f.eks. 1234 56 78903"` er bedre enn `placeholder="1234 56 78903"`.

### Hjelpetekst gir stĆøtte der det trengs

Hjelpetekst brukes for Ć„ gi ekstra kontekst om krav til format eller innhold. Hold den kort, konkret og plassert nĆ¦r feltet. God hjelpetekst kan redusere feil og behovet for feilmeldinger i etterkant.

| GjĆør dette | Ikke dette |
|------------|------------|
| `description="11 siffer uten mellomrom"` | `description="Vennligst fyll inn kontonummeret ditt i det formatet som er korrekt for din kontotype"` |

### Feilmeldinger skal hjelpe, ikke bare varsle

Feilmeldingen skal forklare hva som er feil og hvordan det rettes. Generelle meldinger som "Ugyldig verdi" gir liten verdi. Klare og konkrete tilbakemeldinger gjĆør det enklere Ć„ komme videre.

| Felt | God feilmelding | DĆ„rlig feilmelding |
|------|----------------|-------------------|
| Kontonummer | "Kontonummer mĆ„ ha 11 siffer" | "Ugyldig verdi" |
| E-post | "Oppgi en gyldig e-postadresse, f.eks. ola@sparebank1.no" | "Feil format" |
| BelĆøp | "BelĆøp mĆ„ vĆ¦re mellom 1 og 500 000 kroner" | "Ugyldig belĆøp" |

Komponenten har ingen innebygd validering. Feiltilstand styres utelukkende av innholdet i feilmeldingsfeltet. Sett `errorMessage` i React eller `textContent` pĆ„ `[data-field="error"]` i HTML.

| Ansvar | Hvem |
|--------|------|
| Vise feilmelding og sette `aria-invalid` | Komponenten |
| Annonsere feilmelding til skjermleser | Komponenten |
| Bestemme *nĆ„r* validering skjer | Du |
| Skrive *hva* feilmeldingen sier | Du |

### Validering bĆør ikke forstyrre

Valider nĆ„r brukeren forlater feltet (blur) eller ved innsending, ikke mens de skriver. For tidlig validering kan oppleves som stĆøy og avbryter flyten unĆødvendig.

### Bredden pĆ„ feltet pĆ„virker forstĆ„elsen

Feltets bredde bĆør tilpasses det brukeren forventes Ć„ skrive. Smale felt signaliserer korte verdier som postnummer, bredere felt signaliserer lengre input som navn eller adresse. Variasjon i bredde gjĆør skjemaet lettere Ć„ skanne. Lik bredde pĆ„ alle felt kan gjĆøre det vanskeligere Ć„ orientere seg.

### Prefix og suffix mĆ„ ikke stĆ„ alene

Prefix og suffix gir visuell kontekst, men leses ikke nĆødvendigvis opp av skjermlesere. Labelen mĆ„ inneholde samme informasjon. Et felt med "kr" som prefix bĆør ha en label som "BelĆøp i kroner", ikke bare "BelĆøp".

## Universell utforming

### Hva du selv mĆ„ sĆørge for

- **Meningsfull labeltekst** ā komponenten kobler label til input, men du mĆ„ skrive god tekst
- **Gode feilmeldinger** ā komponenten viser dem og annonserer dem, men du skriver innholdet
- **Riktig `autocomplete`** ā sett `autocomplete="name"`, `autocomplete="email"` osv. pĆ„ felter som ber om personlig informasjon (WCAG 1.3.5)
- **Riktig `type` og `inputMode`** ā `type="email"` for e-post, `inputMode="numeric"` for tall. Gir riktig tastatur pĆ„ mobil og bedre validering.
- **Prefix/suffix i label** ā prefix og suffix er skjult fra skjermlesere. NĆ„r konteksten ikke er Ć„penbar, mĆ„ labelen inneholde samme informasjon: "Rente i prosent", ikke bare "Rente"

### Hva komponenten gjĆør automatisk

NĆ„r du bruker `<ix-field>` eller React-komponenten, settes dette opp for deg:
- **`for`/`id`**: Label kobles til input. Genererer en unik ID hvis input mangler en.
- **`aria-describedby`**: Input peker til description og error-elementet. RekkefĆølgen er meningsfull. Hjelpetekst leses fĆør feilmelding.
- **`aria-live="polite"`**: Settes pĆ„ error-elementet slik at skjermlesere annonserer nye feilmeldinger uten Ć„ avbryte brukeren.
- **`aria-invalid`**: Synkroniseres automatisk med innholdet i error-elementet via MutationObserver.
- **`required`**: Settes pĆ„ input for nativ nettleservalidering. Ingen visuell indikator ā merk valgfrie felt med `(valgfritt)` i labelteksten.

## Hva ix-field gjĆør med DOM-en

`<ix-field>` er en web component som setter opp ARIA-koblinger automatisk nĆ„r den kobles til DOM. Her er hva som skjer ā fra HTML du skriver til HTML etter at `ix-field` har kjĆørt.

**FĆør** (det du skriver):

```html
<ix-field>
  <label>BelĆøp i kroner</label>
  <span data-field="description">Maks 500 000 kroner</span>
  <div class="ix-text-field">
    <div data-field="prefix">kr</div>
    <input inputmode="numeric" />
  </div>
  <span data-field="error"></span>
</ix-field>
```

**Etter** (det som er i DOM etter at `ix-field` har kjĆørt):

```html
<ix-field>
  <label for="ix-field-1">BelĆøp i kroner</label>
  <span data-field="description" id="ix-field-1-description">Maks 500 000 kroner</span>
  <div class="ix-text-field" data-has-prefix>
    <div data-field="prefix" aria-hidden="true">kr</div>
    <input inputmode="numeric" id="ix-field-1"
           aria-describedby="ix-field-1-description ix-field-1-error" />
  </div>
  <span data-field="error" id="ix-field-1-error" aria-live="polite"></span>
</ix-field>
```

Hva som ble lagt til:

| Hva | Hvor | Hvorfor |
|-----|------|---------|
| `for="ix-field-1"` | `<label>` | Kobler label til input ā klikk pĆ„ label gir fokus til feltet |
| `id="ix-field-1"` | `<input>` | Grunnlag for alle ARIA-koblinger. Genereres bare om input mangler en ID |
| `id="ix-field-1-description"` | description-span | NĆødvendig for `aria-describedby`-kobling |
| `id="ix-field-1-error"` | error-span | NĆødvendig for `aria-describedby`-kobling |
| `aria-describedby="... ..."` | `<input>` | Skjermleseren leser description og feilmelding etter label og verdi |
| `aria-live="polite"` | error-span | Skjermleseren annonserer feilmeldinger automatisk nĆ„r de dukker opp |
| `aria-hidden="true"` | prefix-div | Prefix er visuell kontekst ā skal ikke leses opp separat |
| `data-has-prefix` | `.ix-text-field` | Signaliserer til CSS at input-feltet skal miste venstre radius |

NĆ„r error-span fĆ„r innhold setter `ix-field` i tillegg `aria-invalid="true"` pĆ„ input automatisk via MutationObserver. NĆ„r den tĆømmes fjernes attributten igjen.
